# Process Baseline – Sysmon Event ID 1

## Data Source

- Sysmon Event ID 1 – Process Creation
- Collected from:
  - CLIENT01 (primary endpoint)
  - DC01 (domain services)

Primary analysis performed in Splunk.

## Objective

Establish normal process execution patterns in the environment in order to:

- Detect abnormal parent-child relationships
- Identify suspicious command-line usage
- Improve triage speed during incident investigations

Process creation telemetry is one of the highest-value detection sources because nearly all attacker activity eventually results in process execution.

## Methodology

Baseline created by:

1. Querying Sysmon Event ID 1 logs across all hosts
2. Sorting by most frequent parent-child relationships
3. Reviewing command-line arguments for common patterns
4. Identifying administrative vs normal user behavior

## Splunk Queries Used

``` SPL
index=sysmon EventCode=1
| stats count by host ParentImage Image CommandLine
| sort -count
```
This query identifies the most common execution chains in the environment.

```SPL
index="sysmon" EventCode=1 host="CLIENT01"
| eval parent=lower(mvindex(split(ParentImage,"\\"),-1))
| eval process=lower(mvindex(split(Image,"\\"),-1))
| where parent="cmd.exe"
| stats count by parent process
| sort -count
```
This query filters for actual administrative user behavior.

## Observed Parent–Child Process Relationships

Initial baseline analysis revealed that process activity is heavily influenced by Splunk telemetry collection and system-level operations. User-driven activity represents a smaller subset of overall execution volume.

The following execution chains were consistently observed and represent expected Windows behavior.

### User Activity

| Parent Process | Child Process | Behavior |
|----------------|---------------|----------|
| splunkd.exe | python3.13.exe | Splunk modular inputs and scripted data collection |
| splunkd.exe | splunk-powershell.exe | PowerShell-based data collection by Splunk |
| splunkd.exe | splunk-regmon.exe | Registry monitoring component |
| splunkd.exe | splunk-netmon.exe | Network monitoring component |
| splunkd.exe | splunk-MonitorNoHandle.exe | File monitoring processes |

These processes represent normal telemetry collection behavior generated by the Splunk platform. \
Because the lab environment has limited user activity, Splunk processes dominate the baseline.

### System Activity

| Parent Process | Child Process | Behavior |
|----------------|---------------|----------|
| services.exe | svchost.exe | Windows service management |
| winlogon.exe | userinit.exe | User login initialization |

These represent normal OS startup and service operations.

### Interactive User Activity

This distinction is important because interactive command-line usage introduces a secondary execution layer:\
explorer.exe → cmd.exe → child process

| Parent Process | Child Process  | Behavior                                               |
| -------------- | -------------- | ------------------------------------------------------ |
| cmd.exe        | btool.exe      | Splunk troubleshooting and configuration inspection    |
| cmd.exe        | splunk.exe     | Manual Splunk CLI interaction                          |
| cmd.exe        | sc.exe         | Windows service query or modification via command line |
| cmd.exe        | reg.exe        | Registry query or modification                         |
| cmd.exe        | powershell.exe | PowerShell launched from command prompt                |
| cmd.exe        | ipconfig.exe   | Network configuration enumeration                      |
| cmd.exe        | cmd.exe        | Nested command shell execution                         |
| cmd.exe        | splunkd.exe    | Splunk service interaction or restart                  |

Analyzing this layer improves visibility into administrative behavior, script execution, and potential living-off-the-land techniques.

### Security-Relevant Processes

| Parent Process | Child Process | Notes |
|----------------|---------------|------|
| lsass.exe | (rare child processes) | Should almost never spawn processes |
| services.exe | powershell.exe | Requires validation |

If these relationships increase in frequency, they may indicate malicious activity.

## Baseline Bias Consideration

Because this lab environment has minimal user-driven activity, telemetry collection tools (Splunk and Sysmon) generate a large portion of observed processes.

In enterprise environments, user applications such as browsers and productivity tools would generate significantly more process diversity.

Understanding environment-driven baseline bias is important to avoid false positives during detection engineering.

## Analytical Observations

### System Processes Generate High Volume Noise

Processes such as:

- svchost.exe
- services.exe

generate frequent activity and should only be investigated when paired with unusual command-line behavior.

### General Command LIne Pattern Observed

- Standard application launches
- Windows service startup parameters
- Browser execution paths
- PowerShell interactive usage
- Basic command execution

### cmd.exe is the Primary User Execution Parent

The majority of processes spawned from cmd.exe in this lab environment reflect:

- Administrative testing activity
- Splunk configuration and troubleshooting
- Manual system inspection
- Network and service enumeration

Because this is a controlled lab environment, these behaviors are expected.

However, in enterprise environments, command-line–spawned processes carry significantly higher detection value than direct GUI-launched applications.

### PowerShell Usage is Low Volume

PowerShell execution occurs primarily during:

- Lab configuration
- Administrative testing

Because of this low baseline frequency, PowerShell becomes a strong detection signal in this environment.

## Baseline Insight

Separating explorer.exe-spawned processes from cmd.exe-spawned processes highlighted the importance of understanding full parent-child execution chains.

Filtering exclusively for:
```SPL
ParentImage="*\explorer.exe"
```
excludes second-layer executions and may conceal shell-driven activity.

A complete user baseline should account for:
- Direct GUI launches
- Interactive shell launches
- Nested execution chains

This layered approach improves detection engineering accuracy and reduces blind spots during incident investigations.

## Detection Value Derived From Baseline

The baseline enables faster identification of:

- Encoded PowerShell execution
- Suspicious parent-child relationships
- Living-off-the-land techniques

This baseline directly supports the following detection rules:

- Encoded PowerShell detection
- Privilege escalation detection
- Service creation detection

Baselines should be periodically recalibrated to account for environmental changes, software updates, and evolving administrative workflows.

## Detection Rules Derived From Baseline

Process baseline analysis shows low-frequency PowerShell usage and minimal encoded command execution.

Because encoded commands were not observed during baseline activity, they represent a strong detection signal.

Detection focus:

- PowerShell with encoded command flags
- Abnormal parent-child execution chains
- Command-line obfuscation patterns
