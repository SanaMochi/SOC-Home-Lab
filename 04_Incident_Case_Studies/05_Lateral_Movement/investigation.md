# Lateral Movement via PsExec Pattern — Investigation Walkthrough

## Investigator Perspective

This investigation begins from Splunk telemetry on DC01. The attack generated two distinct
events on the target host — a Type 3 network logon (EventCode 4624) and a service installation
(EventCode 7045) — which individually are not unusual but together form the defining signature
of PsExec-pattern lateral movement. The investigation demonstrates how to identify and correlate
these events to reconstruct the attack chain.

Domain Controller compromise is the highest-severity finding in most enterprise environments.
All findings here would be treated as critical priority.

## Step 1 — Identify the Network Logon on DC01

The first indicator is Event ID 4624 with Logon Type 3 on DC01 from an external source address.
Logon Type 3 (Network) is generated by SMB share connections — exactly what `net use \\DC01\ADMIN$`
produces. This is not an interactive desktop login but a credential-authenticated remote network
session, which is the prerequisite step of every PsExec-style attack.

![Raw expanded Event ID 4624 showing Type 3 logon from 192.168.113.20](../screenshots/4624_full.png)

Full raw event detail:

| Field | Value | Significance |
|-------|-------|-------------|
| Time | 03/12/2026 09:14:39.405 PM | T+0 — attack start |
| EventCode | 4624 | Successful logon |
| ComputerName | DC01.corp.local | Target — domain controller |
| Logon Type | 3 | Network logon — SMB/ADMIN$ connection |
| Account Name (New Logon) | Administrator | Highest-privilege domain account |
| Account Domain | CORP.LOCAL | Domain-level authentication |
| Security ID | S-1-5-21-...-500 | Built-in Administrator SID |
| Logon ID | 0xF04D084 | Session pivot key — correlate with other events |
| Elevated Token | Yes | Full admin privileges in this session |
| Impersonation Level | Delegation | Can impersonate on remote systems |
| Logon Process | Kerberos | Domain authentication — not local |
| Authentication Package | Kerberos | Kerberos ticket used, not NTLM |
| Source Network Address | **192.168.113.20** | **CLIENT01 — the attack originated here** |
| Source Port | 55177 | Ephemeral port from CLIENT01 |

The source IP 192.168.113.20 is CLIENT01. This is the critical pivot — it establishes
that an Administrator session was initiated FROM a workstation TO a domain controller,
which is anomalous. Legitimate DC administration should originate from designated admin
hosts, not standard workstations.

The Logon ID `0xF04D084` is a session identifier that can be used to correlate all
activity performed under this logon session. Any subsequent events with the same Logon ID
on DC01 belong to the same attacker session.

## Step 2 — Identify the Service Creation on DC01

Approximately 85 seconds after the network logon, EventCode 7045 appeared in DC01's
System log. This event is generated by the Windows Service Control Manager whenever a
new service is installed. In a PsExec-pattern attack, this is the execution step — the
attacker's command runs inside the service binary path field.

![Raw expanded Event ID 7045 showing SimulatedPsExec service installation](../screenshots/7045_full.png)

Full raw event detail:

| Field | Value | Significance |
|-------|-------|-------------|
| Time | 03/12/2026 09:16:04.652 PM | T+1:25 — 85 seconds after logon |
| EventCode | 7045 | New service installed |
| ComputerName | DC01.corp.local | Target — service installed on DC |
| Service Name | **SimulatedPsExec** | Non-standard name — no vendor context |
| Service File Name | **cmd.exe /c echo lateral_movement_test > C:\Temp\lateral_test.txt** | Raw command — not a service executable |
| Service Type | User mode service | Runs in user space |
| Service Start Type | Demand start | Manually started, not persistent |
| Service Account | **LocalSystem** | Executes as SYSTEM — highest local privilege |
| SID | S-1-5-21-...-500 | Administrator SID — matches the 4624 logon |
| Source | Microsoft-Windows-Service Control Manager | Authoritative service log |

The Service File Name field is the binary path. Legitimate services use dedicated
executables — `cmd.exe /c echo` in a service binary path is an immediate high-confidence
indicator of attacker activity. The service account of LocalSystem means whatever command
runs executes with full SYSTEM privileges on DC01, regardless of who created the service.

The SID in the 7045 event (`S-1-5-21-...-500`) matches the Administrator SID from the
4624 event — confirming both events belong to the same attacker session.

## Step 3 — Run the Basic Detection Query

The targeted detection query confirms both events exist on DC01 within the same time window:

```spl
index=wineventlog host="DC01" ((EventCode=4624 AND Logon_Account_Name=Administrator) OR EventCode=7045)
| table _time host EventCode Logon_Account_Name Logon_Type Service_Name Service_File_Name
| sort -_time
```

![Splunk query showing Administrator Type 3 logons and SimulatedPsExec 7045 on DC01](../screenshots/q2.png)

Returns 6 events in a 5-minute window. The pattern is clear:

| Time | EventCode | Account | Logon_Type | Service_Name | Service_File_Name |
|------|-----------|---------|------------|-------------|-------------------|
| 21:16:14 | 4624 | Administrator | 3 | | |
| 21:16:14 | 4624 | Administrator | 3 | | |
| 21:16:14 | 4624 | Administrator | 3 | | |
| **21:16:04** | **7045** | | | **SimulatedPsExec** | **cmd.exe /c echo lateral_movement_test > C:\Temp\lateral_test.txt** |
| 21:16:04 | 4624 | Administrator | 3 | | |
| 21:14:39 | 4624 | Administrator | 3 | | |

The 4624 at 21:14:39 is the initial ADMIN$ share connection. The 7045 at 21:16:04 is the
service creation. The cluster of 4624 events after the 7045 are generated by the sc.exe
start command and its SMB control channel — each remote sc.exe operation re-authenticates.


## Step 4 — Run the Service-Only Query (Baseline Comparison)

To isolate the service creation and compare against the known-good baseline:

```spl
index=wineventlog host="DC01" EventCode=7045
| table _time host Service_Name Service_File_Name
| sort -_time
```

![Splunk all-time 7045 query showing SimulatedPsExec at top, Azure services below](../screenshots/q1.png)

Returns 34 total service installations on DC01 all-time. SimulatedPsExec is immediately
at the top — the most recent install. The remaining entries are legitimate Azure Connected
Machine Agent services installed on 3/5/26 during lab setup.

The contrast is stark. Every legitimate service has a full executable path referencing
a known vendor directory. SimulatedPsExec has a raw cmd.exe shell command. This is the
difference between a real service and an attacker's execution vehicle.


## Step 5 — Run the Attack Chain Correlation Query

The correlation query combines both event types chronologically to make the attack
sequence visible as a single result set:

```spl
index=wineventlog host="DC01"
| where EventCode=4624 AND Logon_Type=3
| append [search index=wineventlog host="DC01" EventCode=7045]
| sort _time
| table _time EventCode Logon_Account_Name Logon_Type Service_Name Service_File_Name
```

![Splunk correlation query showing Type 3 logons and 7045 in chronological order](../screenshots/q3.png)

Returns 61 events across 15 minutes. The attack chain is visible in chronological order —
the initial Administrator Type 3 logon at 21:14:39, followed by the SimulatedPsExec service
creation at 21:16:04, followed by additional Administrator logons from the sc.exe start attempt.

The 85-second gap between first logon and service creation is consistent with a scripted or
tool-assisted attack rather than manual command-by-command entry — real PsExec typically
executes both steps in under 5 seconds.

## Step 6 — Attacker Terminal Evidence

The CLI output from the attacking session on CLIENT01 confirms the exact sequence executed:

![CLIENT01 PowerShell terminal showing net use, sc.exe create, and sc.exe start commands](../screenshots/cli.png)

```
net use \\DC01\ADMIN$ /user:corp\Administrator  →  The command completed successfully.
sc.exe \\DC01 create SimulatedPsExec binpath= "..."  →  [SC] CreateService SUCCESS
sc.exe \\DC01 start SimulatedPsExec  →  [SC] StartService FAILED 1053
```

`CreateService SUCCESS` confirms the 7045 event is genuine. The `StartService FAILED 1053`
error is expected — cmd.exe is not a persistent service and exits immediately without
completing the SCM registration handshake. The key telemetry (4624 + 7045) was captured
regardless of the start failure.

## Step 7 — Check Defender on DC01

The DC01 Defender timeline was reviewed for the attack window.

![DC01 Defender device timeline showing only background noise during attack window](../screenshots/defender.png)

No lateral movement alert was generated. The only Defender event near the attack time was
an AntivirusReport on services.exe at 21:16:05 — routine background AV scanning on the
legitimate Windows Service Control Manager binary. This is unrelated to the attack and is
present constantly as background activity on any Defender-monitored host.

**Critical analyst note:** The 21:16:05 AV report and the 21:16:04 service creation share
a one-second timestamp window but have no causal relationship. Treating the AV report as
a Defender detection of this attack would be an incorrect finding. Correlation by timestamp
alone is not causation — the action type `AntivirusReport` on a canonical System32 binary
confirms this is background noise, not a detection.

## Step 9 — Key Investigation Questions Answered

**Did the attacker achieve remote execution on DC01?**
The service was successfully created (7045 confirmed) and the start was attempted. The
start failed with error 1053, meaning the cmd.exe payload did not fully execute. In a real
attack using a proper service-compatible payload, execution would have succeeded with
identical telemetry up to that point.

**What credentials were used?**
CORP\Administrator — SID S-1-5-21-...-500, the built-in domain administrator. Kerberos
authentication with an elevated token and Delegation impersonation level.

**Where did the attack originate?**
Source Network Address 192.168.113.20 — confirmed as CLIENT01. This is the lateral movement
pivot point — the attacker controlled CLIENT01 and used it to reach DC01.

**What privilege did the service run as?**
LocalSystem — the highest local privilege account on Windows. Any command in the service
binary path executes as SYSTEM on DC01.

**Was the attacker's activity traceable back to their session?**
Yes. The SID `S-1-5-21-...-500` appears in both the 4624 logon and the 7045 service creation,
confirming both events belong to the same Administrator session. The Logon ID `0xF04D084`
from the 4624 event can be used to pivot for any other activity in that session.

## Step 10 — Recommended Response Actions

1. **Isolate CLIENT01** — the originating host has been used as an active attack platform
2. **Review DC01 completely** — check C:\Temp for staged files, audit all services installed
   in the attack window, review recent AD and GPO changes
3. **Correlate the Logon ID** — search for all events on DC01 with Logon ID 0xF04D084 to
   identify everything done in the attacker's session
4. **Investigate the asmith logon** at 21:16:12 visible in the correlation query — determine
   if this account was also compromised or used for further lateral movement
5. **Reset CORP\Administrator credentials** — this account must be treated as compromised
6. **Hunt domain-wide** for 7045 events with non-standard binary paths across all hosts —
   if DC01 was targeted, other hosts may also have been compromised
7. **Restrict ADMIN$ access** — implement network controls blocking workstation-to-DC SMB
   and enforce Privileged Access Workstations for domain admin activity
